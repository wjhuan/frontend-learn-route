## 简述一下你的脚手架开发流程

1. 基于 lerna 和 pnpm 搭建脚手架项目，pnpm 主要用于管理依赖安装和多 package 间的依赖关系，learn 主要用于发布各个 npm 包以及多个包的版本管理，lerna 已经支持 pnpm，设置`"npmClient": "pnpm" and "useWorkspaces": true in lerna.json`
2. 在 package.json 中增加 bin 命令，也就是脚手架的执行入口文件，在入口文件中增加`#! /usr/bin/env node`,代表用 node 去自动执行这个文件
   > 在用 npm -g 全局安装脚手架时，是把脚手架的包下载 node 的 lib 中的 node-modules 中，在 node 的 bin 目录下配置脚手架命令的软连接，链接到 node modules 中的路径
3. 采用 import-local 来判断优先加载本地脚手架资源否则加载全局的安装的脚手架
4. 使用 commander 完成脚手架命令注册，根据参数 targetPath 参数判断是否加载本地命令，不传则根据命令动态加载对应命令的 npm 包
5. 设计通用的 npm package 类用于缓存下载包以及各种方法，比如对比线上最新版本更新依赖
6. 通过 commander 注册命令 action 动作动态执行命令传入 exec 函数
7. exec 函数根据命令去加载不同的代码去执行，执行采用 node 多进程 spawn 来执行，避免阻塞住进程，通过 stdio: 'inherit',来进行命令行交互，spawn 是用来开启多进程，传入 node -e 命令来执行文件，win32 采用 cmd 来执行命令

8. 单人 git flow 流程

- 本地初始化 git 仓库 `git init`
- 关联远程仓库 `git remote add origin 远程仓库链接地址`
- 推送代码至 master `git push -u origin master`
- 基于 master 分支创建开发分支 `git checkout -b dev/0.0.1`
- 在开发分支修改代码，把新建的分支 push 到远端：`git push origin dev/0.0.1`
- 将本地分支与远端分支关联：`git branch --set-upstream-to=origin/dev/0.0.1`
- 继续修改代码，并提交到远程分支，并且想要部署
- 部署成功之后，合并 dev 到 master，并基于 master 创建 tag 并删除开发分支
  - 在 master 分支运行`git merge dev`
  - `git push origin master`
  - `git tag release/0.0.1`
  - `git push origin release/0.0.1` 或者 `git push --tags`
  - `git branch -d dev/0.0.1` 删除本地开发分支
  - `git push origin --delete dev/0.0.1` 删除远程开发分支并且删除追踪分支

10. 多人 git flow 流程

> 就是在开发过程需要提交代码之前要先 拉取 dev 代码保证 dev 代码是最新提交的，在想要发布时需要先拉取 dev 代码在同步 master 代码，因为要保证发布前的代码要和 master 把持同步

## 重构前端项目需要考虑哪些

1. 理解现有代码：
   分析现有代码结构，了解其架构、模块和依赖关系。
   理解现有功能和业务逻辑。
2. 制定重构策略：
   确定重构的目标和范围，是整体重构还是逐步重构？
   制定计划，确保重构过程不会影响现有功能。
3. 制定测试计划：
   编写单元测试和集成测试，确保重构后的代码功能正常。
   确保在重构过程中保持测试覆盖率。
4. 逐步重构：
   按照计划逐步修改代码结构，使用新的最佳实践替换过时的代码。
   使用可重用组件和模块化的方法来优化代码。
5. 持续集成和部署：
   确保重构后的代码能够顺利地集成到项目中，并能够正常部署。

### 考虑的问题：

1. 性能优化：
   重构是否可以改善项目的性能？
   是否有需要优化的地方，比如减少页面加载时间或提高用户体验？
2. 代码质量：
   重构是否可以提高代码的可读性和可维护性？
   是否可以减少重复代码，并使用更清晰的命名和注释？
3. 安全性：
   重构是否会影响项目的安全性？
   是否需要更新安全措施，比如防止 XSS 或 CSRF 攻击？
4. 兼容性：
   是否需要考虑不同浏览器或设备的兼容性问题？
5. 用户体验：
   重构是否可以改善用户界面和交互体验？
   是否有可以改进的地方，使用户操作更流畅？
6. 团队合作：
   是否需要与团队其他成员协作，确保重构过程顺利进行？
   重构是一个渐进的过程，需要慎重考虑并确保在整个过程中保持良好的沟通和团队合作。

7. 持续集成和部署：
   实施持续集成和持续部署流程，确保代码在合并后自动构建和测试。

### 注意事项：

   及时评估项目进展并调整计划，以适应变化和挑战。
4. 团队合作和支持：
   鼓励团队合作和支持，解决问题时保持积极的态度。
   给予团队成员积极的反馈和认可。
5. 质量和测试：
   确保代码质量，执行必要的测试并进行代码审查。

## 说一说你对 pwa 的理解

> 渐进式 Web 应用程序（Progressive Web Apps，PWA）是一种结合了 Web 和原生应用优点的现代应用程序开发方法。它们利用现代浏览器的功能，让 Web 应用具备类似原生应用的体验。

### PWA 的特点：

1. 可靠性：
   PWA 可以在不稳定的网络环境下工作，能够提供可靠的用户体验。
   利用 Service Workers 可以缓存内容，即使离线状态下也能加载应用。
2. 快速加载：
   通过应用 shell 和缓存策略，PWA 可以实现快速加载，并提供即时的用户响应。
3. 响应式：
   PWA 具有响应式设计，可以适配不同设备和屏幕尺寸，提供一致的用户体验。
4. 类似原生应用的体验：
   PWA 可以通过添加到主屏幕和全屏模式等特性提供类似原生应用的交互体验。
5. 安全性：
   通过 HTTPS 等安全技术，PWA 能够保证数据的安全性，防止恶意攻击。

### PWA 的核心技术：

1. Service Workers：
   允许在后台进行资源缓存和拦截网络请求，提供离线访问和推送通知等功能。
2. Web App Manifest：
   JSON 文件，定义应用的元数据，包括图标、名称、主题色等，影响应用的添加到主屏幕的外观和行为。
3. HTTPS：
   PWA 需要通过安全的 HTTPS 连接来确保数据传输的安全性和用户隐私的保护。

### PWA 的优势：

跨平台：可以在多种设备和平台上运行，无需开发不同版本的应用。
可发现性：PWA 可以被搜索引擎索引，提高应用的可见性。
更新方便：无需手动安装更新，通过网页即可获取最新版本。

## 用户在下单时发现金额和自己计算的有差异，你怎么去排查解决这个问题

检查前端代码：首先审查前端代码，确保计算金额的逻辑正确无误。检查所有相关的数学运算、数据处理、以及金额计算的地方。

确认数据来源：核对前端所使用的数据源，包括用户输入、从后端获取的数据等。确保数据的准确性和一致性。

调试和日志记录：使用调试工具（如浏览器开发者工具）进行实时调试，检查每个步骤的数据处理过程。记录日志以追踪数据的变化，有助于发现问题所在。

与后端交互：如果涉及后端数据，检查与后端的数据交互是否正确。确认后端返回的数据与前端预期一致，并排查可能存在的数据传输或处理问题。

浏览器兼容性：确保代码在各种浏览器中表现一致。有时候特定浏览器的行为可能导致金额计算出现偏差。

边界情况测试：尝试使用不同的输入值和情况进行测试，包括边界情况和特殊情况。例如，输入大额数字、小数、负数等，检查计算结果是否如预期。

版本控制与回滚：如果是最近的更改引起了问题，考虑通过版本控制系统回滚到之前的稳定版本，以确定问题来源。

与团队合作：与团队其他成员合作，特别是后端开发人员或负责数据处理的人员，以确保全面检查和解决问题。

修复和测试：一旦发现问题所在，进行修复并进行全面的测试，确保问题得到解决，并且金额计算与预期一致。

文档和预防措施：在解决问题后，及时更新文档，记录问题及解决方案。同时，考虑采取预防措施，例如添加额外的检查和测试，以避免类似问题再次发生。

## 作为一名资深前端工程师，你平时怎么排查问题

复现问题：首先要能够在自己的环境中复现问题。尝试重现用户报告的问题，这可以帮助确认问题的具体表现和触发条件。

检查控制台日志：查看浏览器控制台的日志，检查是否有错误信息、警告或异常情况。这些信息有助于快速定位问题所在。

使用调试工具：利用浏览器的开发者工具进行调试。检查网络请求、查看元素的样式和结构，以及逐行调试 JavaScript 代码。

代码审查：仔细审查相关代码，特别关注涉及问题区域的逻辑、变量和函数。有时候问题可能隐藏在细微的代码片段中。

数据检查：核对数据的来源和处理过程。确保数据的准确性和一致性，排除数据处理错误可能导致的问题。

跨浏览器测试：检查问题在不同浏览器和设备上的表现。有时候特定浏览器或设备会导致问题，这需要特别留意。

版本控制：使用版本控制系统进行代码对比，查看最近的更改，确认问题是否与某次更新有关。

与团队合作：与团队成员合作，包括后端开发人员、设计师等，共同定位和解决问题。

搜索资源：查阅文档、在线资源、论坛和社区，寻找类似问题的解决方案或经验分享。

逐步排除：通过逐步排除可能性，缩小问题范围。例如，暂时移除某些代码块或功能，以确定问题出现的具体位置。

持续学习：不断学习新的技术和工具，提升自己的排查问题的能力。技术领域不断发展，保持学习态度非常重要。
